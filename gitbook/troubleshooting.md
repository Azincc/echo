# 常见问题与排障

## 1. 登录时报“无法连接到服务器”

优先检查：

- 地址是否写成完整 URL
- 端口是否正确
- 客户端与服务器是否在同一网络或可达
- 反向代理是否正常转发到 Navidrome

正确示例：

- `http://192.168.1.10:4533`
- `https://music.example.com`

## 2. 使用 HTTP 时总会弹安全提示

这是客户端的保护行为，不是异常。

建议：

- 能用 HTTPS 就用 HTTPS
- 只有在可信局域网里才建议继续使用 HTTP

## 3. 新增线路时提示“验证失败”

原因通常是：

- 这个地址实际上指向了另一台服务器
- 当前地址和原音乐库并不是同一份内容
- 地址虽然可访问，但校验过程失败

解决方式：

- 确认多地址确实属于同一个 Navidrome
- 不要把不同服务器混进同一个音乐库

## 4. Embed Service 测试连接失败

最常见的原因有三类：

- 填成了 Navidrome 地址，而不是 Embed Service 地址
- API Key 不正确
- 端口填错了

例如：

- 如果你暴露的是 `5434:8080`，那客户端应该填 `http://<host>:5434`
- 如果你暴露的是 `8080:8080`，那客户端应该填 `http://<host>:8080`

## 5. 离线任务一直卡在“等待中”

优先检查：

- Worker 是否已经启动
- `/work/data/embed.db` 是否可写
- `/music` 挂载目录是否存在且可写
- `NAVIDROME_BASE_URL` 是否能从容器内访问

如果用 Docker Compose，可以先看：

```bash
docker compose logs -f embed-service
```

## 6. 离线任务显示完成，但 Navidrome 里看不到歌

优先检查目录映射是否一致。

当前项目默认会把文件写到容器内：

```text
/music/library/歌手/专辑/xx - 标题.ext
```

如果你把宿主机 `./music-root` 挂载到容器 `/music`，那实际文件会落到：

```text
./music-root/library/...
```

这时 Navidrome 也必须读取同一个 `./music-root`。

## 7. 本地下载后找不到文件

到“下载管理”页顶部先看当前目录提示。

当前实现默认是：

- Android：`Music/Echoes`
- 其他平台：应用文档目录下的 `echo_downloads`

如果目录被手动删改过，可以在下载管理页使用“扫描本地文件”。

## 8. 远程搜索能搜到歌，但不能离线导入

通常是因为：

- 当前音乐库没有配置 Embed Service
- Embed Service 测试连接没有通过
- 你处于本地搜索模式，而不是远程搜索模式

## 9. 哪些问题值得优先导出日志

建议优先导出客户端日志的场景：

- 登录后线路频繁切换
- 歌词或封面获取异常
- 本地下载失败
- 远程试听失败
- 离线导入按钮无响应

## 10. 建议的排障顺序

1. 先确认 Navidrome 本身单独可用
2. 再确认 Echoes 能正常登录和播放
3. 最后再接入 Embed Service

按这个顺序处理，最容易定位问题是在客户端、Navidrome，还是离线导入链路上。
