# 系统架构

## 组件关系

本项目实际由四部分组成：

| 组件 | 作用 | 是否必须 |
| --- | --- | --- |
| Echoes 客户端 | 播放、浏览、搜索、设置、下载管理 | 必须 |
| Navidrome | 主音乐库、Subsonic/OpenSubsonic API、流媒体播放 | 必须 |
| GDStudio Public API | 远程源搜索、试听链接、部分歌词/封面信息 | 仅远程探索需要 |
| `gdstudio-embeded-service` | 把远程歌曲下载到服务器并触发 Navidrome 扫描 | 仅离线导入需要 |

## 两条核心链路

### 1. 日常播放链路

```text
Echoes -> Navidrome -> 音乐文件
```

这一条链路负责：

- 登录与认证
- 首页推荐
- 音乐库浏览
- 歌单、收藏、搜索
- 播放、歌词、封面、缓存
- 客户端本地下载

### 2. 离线导入链路

```text
Echoes -> Embed Service -> GDStudio Public API -> 写入服务器音乐目录 -> 通知 Navidrome 扫描
```

这一条链路负责：

- 搜索远程音源
- 试听远程歌曲
- 把远程歌曲提交到服务器侧下载队列
- 自动写入标签、封面、歌词
- 将文件移动到音乐目录
- 触发 Navidrome 重扫，把新歌纳入音乐库

## “本地下载”和“离线导入”的区别

这是本项目最容易混淆的两个功能。

| 功能 | 数据来源 | 文件落点 | 主要用途 |
| --- | --- | --- | --- |
| 本地下载 | 已经在 Navidrome 音乐库里的歌曲 | 客户端设备本地 | 出门离线听 |
| 离线导入 | 远程搜索到的试听歌曲 | 服务器音乐目录 | 给服务器补歌、补专辑 |

简单理解：

- 如果你要把“服务器里已经有的歌”保存到手机，使用本地下载。
- 如果你要把“服务器里还没有的歌”导入到 Navidrome，使用 Embed Service 离线导入。

## 多地址与自动切换

Echoes 对每个音乐库可以维护多个服务器地址，用于适配：

- 局域网地址
- 公网域名
- 反向代理入口
- 不同网络环境下的可用线路

客户端会根据优先级、可达性和连接状态自动选择线路，并支持在连接失败后自动切换。

## 推荐部署原则

- Navidrome 和 Embed Service 最好部署在同一台主机，或至少能访问同一份音乐目录。
- Navidrome 与 Embed Service 应共享同一份宿主机音乐目录。
- 客户端里填写的 Embed Service URL 必须是 Embed Service 的地址，不是 Navidrome 地址。
- 如果你开启离线导入，建议先把 Navidrome 正常跑通，再接入 Embed Service。
